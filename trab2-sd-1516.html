Goal

The goal of the project is to extend the system created in Project 1 to address the following challenges:

  • Interact with an external service
  • Make communications secure
  • Fault tolerance: replicate data at multiple servers 

Features

1 - Interaction with an external service (6 values)

Create a server (either in REST or SOAP) that act as a proxy to a remote service (e.g. Imgur, Flickr). This server/proxy will allow clients to access albums and photos stored in the account of a user at a remote service.

Note that while in the servers that were implemented on the first project, we consider that a picture with the same name in the same album are the always the same picture (this is still valid in this project for SOAP and REST servers), in Imgur an account can have more than one album with the same name (field title) and more than one picture in the same album with the same name (field title). In this case, you should report to the client different names. For instance, you can report a name for the object (album or photo) that is the concatenation of that object title with its unique identifier used by imgur.

NOTE: the proxy should access the general REST or SOAP APIs provided by the services directly, i.e., you should not use any client library provided by these services.

2 - Secure communications (3 values)

Communications between the clients and the servers must be secure, using secure channels (SSL/HTTPS). Mandatory for REST servers.

Since SSL/HTTPS will only authenticate the server, some provision should be taken to authenticate the clients. For instance, one possible solution is to have a shared password among clients and servers.

3 - Full album replication (5 values)

The contents of an album must be replicated in all servers of the system (excluding those servers that act as a proxy to a remote service).

The replication solution employed by students should provide, at least, the semantics of eventual consistency. Which implies that after the system stops receiving user operations and all servers are synchornized, all servers should have the same final state (i.e, the albuns and photos stored in each server are exactly the same).
Students should also consider special cases where the behavior of the replication system can lead to a user perceived behavior that goes against its expectations. For instance, if a photo is deleted by a user, and no other user uploads it again, the photo should not remain visible in the system.

NOTE: besides updating all servers when a new photo is added or a photo is removed, you must also consider the need to synchronize the state of a server that re-joins the system.

4 - Full album replication + proxy  - optional feature in alternative to 3 (7 values)

This option replaces the previous one by also requiring data to be replicated from and to the external services.
NOTE: you must consider that the state of an album in the external service may change due to direct access to the service, and not throught the service.
SUGGESTION: periodically check if the server state has changed.

5 - Partial album replication - optional feature in alternative to 3 and 4 (7 values)

This option replaces the "Full album replication". and the previously discussed optional feature (4).

The goal of this option is to maintain at least K replicas of a photo in the servers that are running (assuming that K servers are running). To this end, the system may need to create new replicas when some server fails.

6 - Partial album replication + proxy  - optional feature in alternative to 3, 4, and 5 (9 values)

This option replaces feature 3, and options 5 and 6 by requiring also to replicate data from and to the external services.

7 - Up-to-date client interface using Event Dissemination System  - optional feature (2 values)

Use an event dissemination system to update the client interface, when a album is created/removed and when a photo is added/deleted to an album.


Environment

IMPORTANT: The project must be demonstrated in the labs, with servers running in at least two computers, either using existing hardware or student's hardware.

The environment is the same of the project 1.

15% of the final grade of the project is dedicated to the overall integrity and correctness of the system, the written report clarity and quality, and the demonstration of the project.


Written Report:

A written report must be delivered by each group describing their work and implementation. The report should have at most 5 pages (excluding appendix), and be delivered in paper at Secretaria in the Informatics Department. The organization of the report is free, and the students should write their report following a structure that makes its contents clear and easy to follow.

The report must cover the following topics.

  • General description of the work performed by the students, clearly identifying which aspects were completed and fully implemented and those that were not.
  • Limitations of the delivered code.
  • In relation to the interaction with Imgur, the students should clearly identify which endpoints of the Imgur API were used to implement each of the operations supported by the system (list albums, create album, delete album, list pictures in an album, add picture, and remove picture).
  • Still in relation to Imgur, the students should indicate clearly if they resort to some form of caching (at the level of the proxy) and indicate what is the data being cached, and how the cache is managed.
  • Clear explanation of the replication mechanism employed by the students. This description must contain:
    • General textual description (including what entities are responsible for managing the replication of contents and the form of replication implemented).
    • Description of the metadata used to support the replication protocol.
    • The pseudocode of the replication algorithm employed by the students (remember that pseudocode is not java code, hence we expect a clear, clean, and simple description of the algorithm -- penalizations will be applied if the pseudocode presented lacks these properties).
  • In appendix, the report must include the servers (REST and SOAP) APIs that were added to support replication. This should include the signature of these methods and the necessary annotations to make those methods available. This is the only information that should be presented in the appendix, and there is no need to provide the java code that implements these methods.
  • Discussion of the implementation decisions taken by the students, when applicable, discussing these decisions in light of possible alternatives.

The report can also cover aspects related with difficulties felt by the students during the execution of the project or other aspects that the students consider relevant.
